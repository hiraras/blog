## 问题 1

前端方面的整体流程是这样的：

1. 首先获得上传的视频文件
2. 根据文件内容使用加密库例如：spark-md5，生成文件 hash
3. 调用 check 接口获取文件上传情况，如果已经有完整文件则认定为秒传，如果没有后端返回对应文件的已上传部分的分片信息（这里也解释了如何实现断点续传，记录每个分片上传情况，前端根据结果）
4. 设定分片文件的大小，假定为 2M 每个，大小根据服务器承载效率设定
5. 根据分片大小计算出文件拆分后的个数，并使用 slice 方法拆分文件
6. 使用 promise.allSettled 同步上传分片文件，上传数据包括：分片数据，文件 hash，分片索引，分片数量
7. 当有失败的分片，则重新上传
8. 如果所有都成功了调用 success 接口服务端合并文件

其中，文件 hash 的生成为这个过程的关键，因为如果用文件夹作为文件唯一标识可能会产生不同文件被当作同一文件的风险，造成文件拼接错误/文件上传状态错误等问题

缺陷：

1. 需要强调跳过已经上传成功的，只上传还未上传的部分
2. spark-md5 不是加密库而是将文件处理成 hash 序列
3. 不能同时上传所有分片文件，而是要设置最大上传数量，避免一次性大规模接口调用，造成服务端压力陡增
4. 都上传成功之后调用 merge 接口，merge 接口需要校验分片齐全，并且保证合并是幂等的，避免重复合成操作
5. 计算 hash 的时候可以使用 web worker 避免 hash 化过程阻塞主线程

## 问题 2

首先结果肯定是以后端为准，因为用户在什么时候会破坏整个分片上传过程是不可预测的，而最终结果是保留在后端，已经确保了可以确定的部分是正确的
当缓存不一致时，首先重新获取后端的状态，并更新前端的状态，然后针对未上传的分片进行重新上传
当发生错误时，分片数量、每个分片的数据、分片索引都是稳定的，只要 merge 接口判断分片齐全，分片数量和前端对的上，就可以保证不会出错

缺陷：

还需要补充，服务端合并时还要确认每个分片的完整性(可能写入了错误数据，操作是可以验证分片的大小)

## 问题 3

同一文件在相同的分片策略下拆分出来的结果是一样的，后端需要在 save 接口保存文件时，判断是否已经存在，避免重复上传，并用事务锁避免同时保存同一个分片文件
前端调用 merge 的时机还是根据分片上传的结果调用，因为真正耗时的工作在 save 接口，它已经做了相应的工作了，可以不必重复执行，后端倒是可以根据两次 merge 状态的执行情况返回不同状态码，前端根据状态吗进行不同提示
当然客户端可以根据 fileHash 判断是否重复上传了同一文件，直接进行阻拦和提示

缺陷：
客户端判断成功可能根据每个分片的请求来，而服务端可能会打标记，两者可能会发生分歧，还是以后端的为准，所以结果判断还是要以后端返回为准

## 问题 4

1. ui 层面上，当第一个请求发生了，应该立即设定提交按钮的状态未 disabled，避免重复发送
2. 如果没有这一步的拦截，可以在每次发送文件的时候取消上一次的请求，使用 AbortController
3. 最后我认为当后端保证了 merge 接口对分片数据、分片数量、文件 hash 等数据的校验，并实现了合并操作是幂等的话，多次执行 merge 的消耗其实也是很小的，可以直接根据 merge 接口返回的状态进行提示或其他操作
