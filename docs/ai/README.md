下面是一套把「技术框架 / 功能模块 / 业务流程」交给 AI 时，**阶段、顺序和载体（Skill / Rule / 文档）**怎么配合，才能让成品尽量贴近你要的样子。

## 核心原则：先锁边界，再拆任务，最后才写细节代码

AI 最怕的是：**框架未定就写页面、流程没说清就写接口**。所以要把「不可变约束」尽量在前面固定下来，并在整个项目里**可重复引用**（Skill、Rule、简短规格），而不是只在某一两条聊天里说过一次。

---

## 各阶段提供什么、用什么形式

### 1. 立项与约束（第 0 步，一次性）

**提供：**

-   目标用户与场景一句话
-   必须支持的平台（仅 Web / 是否要移动端适配）
-   合规与安全边界（登录方式、数据能否出境等，若有）

**载体：**

-   **Cursor Rule（`.cursor/rules`）**：写「永远遵守」的项目级约束（技术栈大方向、目录约定、禁止事项）。
-   可选：**Skill**：若你希望 AI 在「每次生成文档/接口说明」时都按同一模板输出，可以用 Skill 固化模板。

这一阶段不要求极其细，但要避免后面全盘推翻。

---

### 2. 技术框架选型（紧接着第 0 步）

**提供：**

-   前端框架（如 React/Vue）、构建工具、UI 库、状态方案
-   后端若需要：运行时、框架、数据库、ORM
-   部署目标（Vercel / 自建 Docker / 静态托管等）

**载体：**

-   **Rule**：把「官方栈」写死，避免 AI 混用两套 UI 库或两套路由方案。
-   简短 **README 或 `docs/architecture.md`**（可选）：一页纸的技术选型理由，方便你和 AI 对齐；不必很长。

**时机：** 在写第一个业务页面之前就要定；否则后面迁移成本全算在你的对话轮次里。

---

### 3. 业务流程（先于「功能列表」细化）

**提供：**

-   主流程：谁、在什么入口、经过哪些步骤、成功/失败怎样
-   分支：权限差异、异常（支付失败、审核驳回等）
-   与外部的交互顺序（若涉及）

**载体：**

-   **Skill** 很适合：例如「根据我提供的业务流程，输出用户故事 + 验收标准 + 接口草案」这类**固定产出格式**。
-   辅以 **Mermaid 流程图** 或分步骤列表放在规格文档里，AI 实现时可反复 `@` 引用。

**顺序说明：** 业务流程应早于「零散功能点」详述。否则容易出现页面都有了但流程接不上的情况。

---

### 4. 功能模块（在流程清晰之后拆解）

**提供：**

-   模块划分（例如：账号、内容、订单、后台）
-   每个模块：核心页面/接口、与其它模块的依赖
-   MVP 范围 vs 后续迭代（写清楚「第一期不做」）

**载体：**

-   **Rule**：模块边界和命名约定（例如路由前缀、`features/` 目录结构）。
-   **Skill**：若你经常让 AI「按模块生成 CRUD + 权限检查」，可用 Skill 规定清单（字段、校验、错误码）。

---

### 5. 数据与接口契约（实现前几轮）

**提供：**

-   核心实体、字段类型、关系
-   API 风格（REST 路径约定、错误格式）
-   鉴权方式（Cookie/JWT、角色）

**载体：**

-   小型 **OpenAPI 片段**、表格或 TypeScript `types`，放进仓库并由 Rule 指向「以仓库里的类型为准」。  
    这样比口头描述更不容易漂移。

---

### 6. UI/UX 期望（可与功能并行，但要在堆界面之前）

**提供：**

-   设计风格关键词或参考站
-   布局习惯（侧栏/顶栏、表单密度）
-   无障碍或国际化是否要第一期做

**载体：**

-   Rule 里写「组件库优先、禁止内联大段魔法样式」之类原则；
-   具体设计 tokens 若固定，可写在主题配置并由 Rule 引用。

---

## Skill vs Rule：怎么选

| 目的                                                                               | 更适合                                                                   |
| ---------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **任何对话都要遵守**（栈、目录、安全、禁止混库）                                   | **Rule**                                                                 |
| **某类重复任务要固定输出结构**（写接口文档、按模块生成页面骨架、按流程写测试用例） | **Skill**                                                                |
| **一次性、很长的背景**（整份 PRD）                                                 | 仓库里的 **文档文件**，聊天里 `@` 引用，必要时摘要进 Rule 的「必读路径」 |

Skill 不是 Rule 的替代品：Rule 管「底线」，Skill 管「这类任务怎么做才符合我」。

---

## 实操顺序小结（给你一条可直接照做的流水线）

1. **Rule**：项目目标 + 技术栈 + 目录与命名 + 禁止事项。
2. **文档或 Skill**：业务流程（ + 简易流程图）。
3. **文档**：功能模块与 MVP 边界。
4. **类型/接口契约** + Rule 里写「实现以仓库类型为准」。
5. 再进入具体页面与接口实现；大功能点用 **Skill** 统一产出格式（例如每个模块必须含哪些文件、哪些测试）。

---

## 常见踩坑（刻意避开能明显提升满意度）

-   **只在第一轮说了技术栈，后面长对话里 AI 遗忘**：把栈写进 Rule，而不是依赖记忆。
-   **功能清单很长但没有流程**：先流程后清单。
-   **口头描述字段**：容易和实现不一致，用类型或表格锁定。
-   **没有 MVP 边界**：AI 会自作主张加功能或反过来漏关键路径，边界要写进文档并由 Rule 指向。
