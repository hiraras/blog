# AI 开发常见术语通俗指南

这份文档面向入门和进阶开发者，整理了 AI 开发中高频出现的名词（如 LLM、RAG、MCP 等），并说明：

- 这些名词是什么
- 它们在项目里的作用
- 它们通常怎么工作
- 它们之间如何协作

---

## 一、常见术语与作用（通俗版）

### 1) LLM（Large Language Model，大语言模型）

- **是什么**：一个“超大号文本预测器”，通过海量语料训练，能理解并生成自然语言。
- **作用**：回答问题、写代码、总结文档、生成内容、做对话。
- **通俗理解**：像一个读过很多资料、会组织语言的“智能助手大脑”。

### 2) Prompt（提示词）

- **是什么**：你发给模型的指令和上下文。
- **作用**：决定模型“做什么、怎么做、输出什么格式”。
- **通俗理解**：给 AI 的任务说明书。

**补充问答（你刚刚关心的两个问题）**

- **问：Prompt 只包含用户最后一句吗？会不会每次都把全部历史原样重发？**
    - 不只最后一句。真实请求通常包含系统规则、开发者规则、当前用户输入，以及被保留的历史上下文（可能是原文，也可能是摘要）。
    - 不是永远“机械全量重发”。工程里常用截断、摘要、检索注入等方式控制上下文长度，避免请求越来越臃肿。

- **问：上下文压缩/摘要会消耗 token 吗？**
    - 会。只要调用模型来做摘要，就会消耗输入和输出 token。
    - 但这样做通常更省整体成本：先花少量 token 压缩，再减少后续多轮请求的历史 token 开销。

### 3) Token（词元）

- **是什么**：模型处理文本的最小单位（不一定是一个完整单词）。
- **作用**：模型的输入输出都按 token 计费和计长。
- **通俗理解**：AI 读写文本时的“字数单位”。

### 4) Context Window（上下文窗口）

- **是什么**：模型一次能“记住并处理”的最大 token 量。
- **作用**：限制你一次能塞给模型多少资料、历史对话和代码。
- **通俗理解**：AI 的“短时工作记忆容量”。

### 5) Embedding（向量化表示）

- **是什么**：把文本/代码转换成向量，让机器可计算“语义相似度”。
- **作用**：语义检索、推荐、RAG 召回。
- **通俗理解**：把文字变成“坐标点”，方便找“意思接近”的内容。

### 6) Vector Database（向量数据库）

- **是什么**：存储 embedding 并支持相似度检索的数据库。
- **作用**：快速找出和问题最相关的知识片段。
- **通俗理解**：AI 的“语义搜索引擎仓库”。

### 7) RAG（Retrieval-Augmented Generation，检索增强生成）

- **是什么**：先检索资料，再把资料喂给 LLM 生成答案。
- **作用**：降低幻觉、引入私有知识、提升准确性和时效性。
- **通俗理解**：先“查资料”，再“作答”，不是只靠模型记忆硬答。

### 8) Fine-tuning（微调）

- **是什么**：在基础模型上继续训练，让其更适配特定任务或风格。
- **作用**：提升领域能力、一致性、格式稳定性。
- **通俗理解**：给通用大脑做“职业培训”。

### 9) Inference（推理）

- **是什么**：模型接收输入后，生成输出的过程（即线上调用阶段）。
- **作用**：提供实际服务结果（回答、分类、生成代码等）。
- **通俗理解**：AI 真正“上班干活”的阶段。

### 10) Temperature（温度）

- **是什么**：控制输出随机性的参数。
- **作用**：低温更稳定保守，高温更发散有创意。
- **通俗理解**：调节 AI “严谨”还是“脑洞大开”。

### 11) Hallucination（幻觉）

- **是什么**：模型输出看似合理但不真实/无依据的信息。
- **作用（风险）**：会导致错误决策、错误代码、错误事实。
- **通俗理解**：AI 一本正经地“编答案”。

### 12) Agent（智能体）

- **是什么**：基于 LLM、能自主分解任务并调用工具的一类系统。
- **作用**：执行多步骤任务，如搜索、写代码、跑命令、生成报告。
- **通俗理解**：不仅会说，还会“动手做事”的 AI 助手。

### 13) Tool Calling（工具调用）

- **是什么**：模型按约定格式调用外部工具（搜索、数据库、API 等）。
- **作用**：突破“只能聊天”，连接真实世界数据与操作能力。
- **通俗理解**：AI 借助手脚去查、算、改、执行。

### 14) Function Calling（函数调用）

- **是什么**：LLM 输出结构化参数，触发后端函数执行。
- **作用**：把自然语言请求转成可执行程序动作。
- **通俗理解**：AI 把“人话”翻译成“程序命令”。

### 15) MCP（Model Context Protocol）

- **是什么**：让模型以统一协议连接外部数据源和工具的标准接口。
- **作用**：减少每个工具重复对接成本，便于扩展生态。
- **通俗理解**：AI 工具接入的“USB 标准接口”。

### 16) Guardrails（护栏）

- **是什么**：约束和校验机制（安全、合规、格式、内容策略）。
- **作用**：降低越权、违规、错误输出风险。
- **通俗理解**：给 AI 系统装“安全带 + 规则红线”。

### 17) Eval（评测）

- **是什么**：用一套测试集和指标评估 AI 系统质量。
- **作用**：验证改动有没有让系统更好，而不是靠感觉。
- **通俗理解**：AI 系统的“单元测试 + 考试评分”。

### 18) Latency（延迟）

- **是什么**：从请求到返回结果的时间。
- **作用**：直接影响用户体验与系统吞吐。
- **通俗理解**：AI 回答“快不快”。

### 19) Throughput（吞吐量）

- **是什么**：单位时间内系统处理请求的能力。
- **作用**：决定并发承载能力与成本效率。
- **通俗理解**：AI 系统“每分钟能接多少单”。

### 20) Caching（缓存）

- **是什么**：把高频结果临时保存，避免重复计算。
- **作用**：降成本、降延迟、提高响应速度。
- **通俗理解**：常见问题先记住，下次直接给答案。

### 21) Rules（规则）

- **是什么**：对 AI 行为的长期约束配置（如风格、边界、禁止项、流程要求）。
- **作用**：让模型在不同会话中保持一致行为，减少“这次记得、下次忘了”。
- **通俗理解**：AI 的“固定工作制度”。

### 22) Skills（技能）

- **是什么**：可复用的任务能力模块，通常包含触发条件、步骤模板、输入输出约定。
- **作用**：把常见任务（如写 PR、做代码审查、生成测试）标准化，提升稳定性和效率。
- **通俗理解**：AI 的“可插拔工作手册”。

### 23) Spec（规格说明）

- **是什么**：对需求、接口、数据结构、验收标准的明确约定文档或结构化定义。
- **作用**：把“模糊需求”变成“可实现、可验证”的工程目标，降低理解偏差。
- **通俗理解**：项目的“施工图纸 + 验收清单”。

### 24) Skill Spec（技能规格）

- **是什么**：专门描述某个 Skill 的目标、触发方式、执行步骤、边界与失败处理的规范。
- **作用**：保证技能可维护、可复用、可审计，避免同名技能行为漂移。
- **通俗理解**：给 AI 技能写的“说明书和操作规程”。

### 25) Transformer（模型核心架构）

- **是什么**：当前主流大模型使用的神经网络架构，核心机制是注意力（Attention）。
- **作用**：高效建模长距离上下文关系，是 LLM 能理解和生成复杂文本的基础。
- **通俗理解**：大模型“大脑电路”的主流设计图。

### 26) LangChain

- **是什么**：用于构建 LLM 应用的开发框架，提供提示模板、工具调用、检索链路等组件。
- **作用**：加速搭建“模型 + 检索 + 工具 + 记忆”的应用流程。
- **通俗理解**：AI 应用开发的“组件工具箱”。

### 27) LangGraph

- **是什么**：LangChain 生态中的图编排框架，用节点和边定义状态化、多步骤 Agent 流程。
- **作用**：适合复杂工作流（循环、分支、人工介入、可恢复执行）。
- **通俗理解**：给 AI Agent 画“流程图并可运行”。

### 28) Dify

- **是什么**：面向企业与开发者的 LLM 应用开发平台（可视化工作流 + 后端能力）。
- **作用**：快速搭建聊天机器人、RAG 应用、Agent 流程和多模型应用。
- **通俗理解**：低代码/可视化的 AI 应用搭建平台。

### 29) Coze

- **是什么**：面向 Bot/Agent 构建与发布的平台，强调插件、工作流和多渠道分发。
- **作用**：让开发者更快创建可调用工具的智能体并上线到不同渠道。
- **通俗理解**：偏“Bot 生产和分发”的 AI 平台。

### 30) Harness（AI 评测/测试基座，广义）

- **是什么**：用于运行、比较和回归验证模型/Agent 行为的一套测试执行框架与环境。
- **作用**：把“AI 效果验证”工程化，支持自动化评测、回归对比、稳定性追踪。
- **通俗理解**：AI 系统的“测试台 + 压测台 + 对比台”。

## 二、这些技术各自的工作过程

### A. LLM 的基本工作过程

1. 用户输入 Prompt。
2. 文本被切成 token。
3. 模型基于上下文概率逐步预测下一个 token。
4. 不断生成直到结束条件满足（如输出长度上限或停止符）。
5. 返回自然语言或结构化结果。

### B. RAG 的工作过程

1. 用户提问。
2. 问题被向量化（embedding）。
3. 在向量数据库中检索相似文档片段。
4. 将“问题 + 检索内容”拼成增强 Prompt。
5. LLM 基于这些资料生成答案（可附来源）。

### C. Tool/Function Calling 的工作过程

1. 模型判断“是否需要外部工具”。
2. 若需要，输出工具名和参数（结构化）。
3. 后端执行工具（查库、调 API、运行程序等）。
4. 将工具返回结果回填给模型。
5. 模型整合信息，输出最终回答。

### D. MCP 的工作过程（简化）

1. AI 客户端连接 MCP Server。
2. MCP Server 暴露可用资源和工具清单。
3. 模型按统一协议请求具体资源/工具。
4. Server 返回标准化结果。
5. 模型继续推理并生成最终输出。

## 三、典型协作流程（从请求到结果）

下面是一个常见“AI 应用生产链路”：

1. **用户发起请求**
    - 例如：“帮我总结这个项目，并给出重构建议。”

2. **Agent 进行任务规划**
    - 拆分子任务：读文档、查代码、调用测试工具、汇总结论。

3. **RAG 检索知识**
    - 从知识库检索相关设计文档、历史记录、接口说明。

4. **LLM 推理与生成**
    - 在检索内容基础上生成解释、方案、代码草稿。

5. **Tool/MCP 执行外部操作**
    - 调接口、查数据库、执行命令、读取文件等。

6. **Guardrails 校验输出**
    - 检查敏感信息、格式、事实引用、权限合规。

7. **返回结果并记录日志**
    - 输出给用户，同时记录指标（延迟、成功率、token 成本）。

8. **Eval 持续改进**
    - 用评测集回归测试，优化 prompt、检索、模型参数与流程。

## 四、一个更直观的“角色分工图”（文字版）

- **LLM**：负责“思考和表达”
- **RAG**：负责“找资料”
- **向量库**：负责“按语义快速检索”
- **Tool / MCP**：负责“动手执行和对接系统”
- **Agent**：负责“统筹多步骤任务”
- **Guardrails**：负责“安全与质量底线”
- **Eval**：负责“持续验收和优化”

你可以把它理解成一个团队：

- LLM 是主讲人
- RAG 是资料员
- 工具系统是执行员
- Agent 是项目经理
- Guardrails 是审计/风控
- Eval 是质检部门

---

## 五、落地建议（实战优先级）

如果你刚开始做 AI 项目，建议按这个顺序建设：

1. 先把 **Prompt + 基础调用链路** 跑通。
2. 再上 **RAG**，解决“知识准确性和可追溯”问题。
3. 接入 **Tool/MCP**，让系统具备真实执行能力。
4. 补上 **Guardrails + Eval**，保证可上线和可迭代。
5. 最后再考虑 **微调**，用于特定场景深度优化。

---

## 六、总结

AI 开发不是只接一个 LLM 接口，而是一个“模型 + 检索 + 工具 + 安全 + 评测”的系统工程。  
理解这些术语之间的关系，有助于你从“会调用模型”升级到“能构建稳定可用的 AI 产品”。

## 七、补充问答（Q/A）

**Q1：AI 开发里说的“记忆大小限制”是什么？长周期、多会话项目怎么解决？有什么省 token 技巧？**  
**A：**“记忆大小限制”通常指模型的上下文窗口（context window）限制，即一次请求中模型能处理的总 token 上限。这个上限包含系统提示、历史对话、当前输入和模型输出预算。长周期项目和多会话并行时，问题会更明显，因为会话之间默认不共享短期上下文，且历史越长越容易触顶。  
可行做法是把“记忆”外置：在项目内维护一个统一上下文文档（如 `AI_CONTEXT.md`），沉淀目标、约束、决策、进展、风险和下一步；每次新会话先读它、结束时再增量更新。再配合里程碑拆分和 ADR（架构决策记录），可以显著降低遗忘与重复沟通。  
省 token 的实用技巧包括：只提供最小必要上下文（路径、函数、关键报错片段），优先给 diff 而非整文件，要求结构化短答，按阶段执行（分析/修改/验证），避免重复粘贴背景，并尽量做“局部更新”而非“整段重写”。

**Q2：让 AI 把已收到内容写到文件里，这个过程会消耗 token 吗？还是客户端自带功能不耗 token？**  
**A：**会消耗 token，但要区分两层：

1. 模型理解指令、组织文本、生成补丁或内容，这部分一定消耗 token；
2. 客户端/IDE 执行“写入磁盘”这个动作本身通常不按模型 token 计费。  
   所以总体仍有消耗，且当内容很长、反复改写整文件时会明显增大。优化方式是固定模板、指定精确更新位置（只改某一节）、采用增量追加、减少重复改写。

**Q3：RAG 里“先检索再喂给 LLM”，是不是代表用大模型时都会自动检索资料？**  
**A：**不一定。普通调用大模型时，很多场景只用模型参数知识和当前对话上下文，并不会自动查外部资料。RAG 是显式增加了“外部检索”步骤，常用于私有知识、最新信息和需要可追溯来源的场景。

**Q4：外部知识是否包含网上最新新闻资讯？**  
**A：**可以包含，但取决于系统是否接入联网检索能力。若已接入搜索/网页抓取工具，可把最新新闻作为外部知识参与回答；若未接入，则只能依赖模型已有知识和用户提供资料，时效性会受限。

**Q5：实际使用时，怎么触发模型去检索外部知识？**  
**A：**常见有三种方式：

1. 固定策略：每次提问都先检索再回答；
2. 规则触发：涉及时效性或专业领域时才检索；
3. 工具调用：给模型开放检索工具，让模型按需调用。  
   落地上就是配置“数据源 + 检索工具 + 触发策略”，由应用编排层执行。

**Q6：RAG 是否就是“用户问题 -> API 检索文档 -> 问题和文档一起发给 LLM -> 输出回答”？**  
**A：**是的，这个理解很准确。可以把它看成在 API/编排层做了一次“检索、筛选、拼装上下文”的组装，然后再交给 LLM 生成最终答案。

**Q7：如果还要做筛选和拼装，工作量会不会没减少？能不能交给 AI 总结？不同功能是不是要做不同 API？**  
**A：**可以把大量“重排与总结”交给 AI，但不建议完全放手。建议由程序负责确定性部分（权限、边界、来源约束、结果校验），由 AI 负责语义总结与表达。  
API 设计上也不必每个功能单独做一个接口，更常见是“统一入口 + 场景路由 + 可插拔检索策略/工具链”，这样复用更高、维护更轻。

分层策略

1. Orchestrator（编排层）：统一入口，决定走哪条链路
2. Retriever（检索层）：不同知识源插件化（文档库、工单库、数据库）
3. LLM（生成层）：负责总结与回答
4. Guardrails（约束层）：权限、引用、敏感信息、格式校验

## 八、补充 Q/A：从自然语言到模型回答（修正版）

**Q：我这样理解对吗？用户的自然语言先转成 token 和向量；模型再按“问题 + 已生成内容”逐步预测下一个 token；最后把 token 还原成自然语言输出。**

**A：对，这个理解已经非常接近模型真实工作方式。可以用三步来记：**

1. **输入编码阶段（读懂问题）**  
   用户输入会先被切成 token，并映射成向量表示。  
   这里不是一次性“硬判断这是哪个领域”，而是模型在后续多层计算中逐步形成对意图、语境和任务方向的理解。

2. **自回归生成阶段（逐步写答案）**  
   模型每次只生成一个 token，并且每一步都会基于：
    - 原始问题
    - 已生成的全部 token  
      重新计算下一 token 的概率分布（softmax 后采样或选最大概率）。  
      这一步会循环，直到遇到停止条件（如结束符或长度上限）。

3. **输出解码阶段（还原成人话）**  
   生成完成后，token 序列被解码成自然语言，返回给用户。

**一个更形象的例子：**  
问题：`帮我写一句今天的天气播报`

- 第一步先理解任务是“天气播报句式”，不是问定义或写代码。
- 生成时可能先输出 `今天`，接着模型会在词表里给很多候选打分，比如 `天气`、`气温`、`有`、`是` 等。
- 若当前上下文下 `天气` 概率更高，就先输出 `天气`；下一步再基于 `今天 天气` 继续预测，例如 `晴朗`、`不错`、`有雨` 等。
- 最终逐 token 形成一句完整话，比如：`今天天气晴朗，体感舒适，适合外出。`

**关键纠偏：**  
这个过程不是“去历史语料里逐条检索最像句子”，而是“用训练得到的参数，在当前上下文下实时计算最可能的下一个 token”。
